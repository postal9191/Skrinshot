# Техническое задание

## Проект

Desktop-приложение для скриншотов и записи экрана с загрузкой файлов на локальный сервер в одной сети.

## 1. Общая информация

### 1.1 Назначение проекта

Необходимо разработать desktop-приложение для Windows, предназначенное для:

- создания скриншотов экрана;
- записи видео экрана;
- базового редактирования скриншотов;
- локального сохранения файлов;
- загрузки файлов на сервер в локальной сети;
- получения ссылки или пути к загруженному файлу.

Приложение должно быть по UX и сценарию использования максимально приближено к классическому «Скриншотеру»: быстрый вызов по горячей клавише, захват области, мгновенное сохранение, копирование и отправка.

### 1.2 Основная задача

Создать удобный, быстрый и стабильный инструмент для ежедневной работы внутри локальной сети без зависимости от облачных сервисов.

### 1.3 Целевая платформа

- Windows 10
- Windows 11

### 1.4 Язык интерфейса

- Русский по умолчанию

## 2. Функциональные требования

### 2.1 Функции скриншотов

Приложение должно поддерживать:

- создание скриншота всего экрана;
- создание скриншота выбранной области;
- создание скриншота активного окна;
- корректную работу с несколькими мониторами;
- корректную работу при DPI scaling Windows;
- сохранение скриншотов в форматах PNG и JPG;
- настройку качества JPG;
- копирование скриншота в буфер обмена;
- автоматическое или ручное сохранение файла.

### 2.2 Постобработка скриншота

После создания скриншота пользователь должен иметь возможность открыть встроенный редактор и выполнить аннотацию:

- стрелка;
- линия;
- прямоугольник;
- эллипс;
- текст;
- маркер или подсветка;
- размытие или скрытие конфиденциальных областей.

### 2.3 Функции записи видео

Приложение должно поддерживать:

- запись всего экрана;
- запись выбранной области;
- сохранение записи в MP4;
- настройку FPS, например 15, 30, 60;
- настройку качества или битрейта;
- отображение индикатора записи;
- запуск и остановку записи по горячим клавишам;
- опционально паузу и продолжение записи;
- опционально запись системного звука и микрофона.

### 2.4 Загрузка файлов на локальный сервер

Приложение должно:

- загружать скриншоты и видео на локальный сервер в той же сети;
- работать с HTTP API;
- отправлять файлы через `multipart/form-data`;
- передавать метаданные файла;
- получать в ответ JSON с информацией о результате;
- показывать прогресс загрузки;
- сохранять локальную очередь файлов при недоступности сервера;
- выполнять повторную отправку автоматически или по команде пользователя;
- выдавать ссылку или путь на загруженный файл;
- позволять копировать ссылку в буфер обмена.

### 2.5 История файлов

Приложение должно хранить историю последних действий:

- дата и время создания;
- тип файла;
- локальный путь;
- статус отправки;
- ссылка на сервере, если файл загружен;
- размер файла.

### 2.6 Настройки

Необходимо реализовать экран настроек со следующими параметрами:

- папка локального сохранения;
- адрес локального сервера;
- Bearer Token для авторизации, если используется;
- автоматическая загрузка после создания;
- формат изображений;
- качество изображений;
- параметры видео;
- поведение после захвата;
- шаблон имени файла;
- настройка горячих клавиш;
- автозапуск вместе с Windows;
- тема интерфейса, если реализуется.

## 3. Требования к UX/UI

### 3.1 Основной сценарий работы

Приложение работает в фоне и располагается в системном трее.

Пользователь должен иметь возможность:

- вызвать захват области по горячей клавише;
- быстро создать скриншот;
- сразу сохранить, скопировать, отредактировать или отправить файл;
- начать и остановить запись экрана;
- открыть историю файлов;
- открыть настройки;
- получить ссылку на загруженный файл.

### 3.2 Требования к интерфейсу

Интерфейс должен быть:

- минималистичным;
- быстрым;
- понятным без обучения;
- визуально современным;
- ориентированным на минимальное число действий.

### 3.3 Tray-режим

Необходимо реализовать:

- иконку в системном трее;
- контекстное меню tray;
- быстрые действия из tray;
- скрытие главного окна без завершения приложения.

## 4. Требования к серверной части

### 4.1 Назначение сервера

Сервер должен принимать файлы от desktop-клиента по локальной сети, сохранять их и возвращать информацию о загруженном файле.

### 4.2 Требуемые API endpoints

- `POST /api/upload/image`
- `POST /api/upload/video`
- `GET /api/files/:id`
- `GET /api/health`

### 4.3 Требования к API

Сервер должен:

- принимать `multipart/form-data`;
- сохранять файлы в структурированную директорию;
- генерировать идентификатор файла;
- отдавать JSON-ответ с полями:
  - `success`;
  - `fileId`;
  - `fileName`;
  - `url`;
  - `size`;
- поддерживать базовую защиту через Bearer Token как опцию;
- корректно обрабатывать ошибки;
- иметь проверку состояния через health endpoint.

### 4.4 Пример JSON-ответа

```json
{
  "success": true,
  "fileId": "abc123",
  "fileName": "screenshot_2026-03-13_10-25-11.png",
  "url": "http://192.168.1.100:8080/files/abc123",
  "size": 245821
}
